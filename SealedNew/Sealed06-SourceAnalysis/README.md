# 06 源码分析

## 分析目标

这里不直接贴整段 .NET 源码，而是用“框架常见写法的简化实现”来分析 `sealed` 的设计动机。

## 观察角度

1. 框架通常先定义抽象基类或接口
2. 再提供一个或多个具体实现类
3. 当某个实现已经表达完整、且不希望外部继承篡改时，会使用 `sealed`

## 为什么很多框架源码这么做

- 公共抽象负责扩展
- 具体实现负责落地
- 最终实现不一定要继续开放继承
- 这样能减少子类破坏不变量的机会

## 源码阅读方法

阅读到 `sealed` 时，重点问三个问题：

- 这个类型是不是一个最终实现？
- 如果允许继承，会不会破坏业务不变量？
- 框架真正希望开放的是“接口扩展”还是“继承扩展”？
